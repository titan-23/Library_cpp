# Action compose（辺縮約）メモ

差分更新ビームサーチの抜本改善案のひとつ。global-LCA より下にできる
**内部一本道**を、1手ずつ apply する代わりに「合成した1手」に畳む。

## 効く条件（2つとも必要）

1. 位相条件：global-LCA より下に*長い内部単一子チェーン*が存在する。
   生存木が「分岐は時々、間は非分岐が長い」形。毎手分岐する問題は対象外。
2. 代数条件：action がモノイドを成し、`compose(a1..aL)` の apply が
   L·a より十分安い。理想 O(1)/O(log)。逆操作も安いこと。

両方そろうと、その鎖の毎世代 2L·a が 2·O(1) に落ちる。

## checkpoint に対する優位

compose は a/s/L の閾値推定が不要。合成オブジェクトは小さく State コピーも
メモリ膨張も無いので「s 払う価値あるか」の判断自体が消える。Action が
compose を出せるなら常に縮約して損しない。閾値問題が無いのが robust な点。

## 抽象例（圧縮できる action）

- 置換：chain = 置換の積 → 1置換。逆も置換。
- アフィン x↦Ax+b：chain = 1アフィン。
- 行列作用 v↦M_a v：chain = 行列積／累乗。
- 区間 add/assign の遅延タグ：タグが合成法則を持つ。
- 加法/カウンタ：chain = 変位の総和（1つの数）。

共通構造：L手の効果を**サイズ一定の1オブジェクト**に畳め、適用も逆も安い。

## 具体例（効く）

- 回廊を動くエージェント：重い状態は位置/占有、分岐は方向のみ。一本道区間は
  単一の平行移動 (dx,dy) に合成。状態移動 O(L)→O(1)。
- オートマトン上の文字列 DP：action=文字=遷移行列。強制文字の連続が行列積1つ。
- 置換列を出力する問題（ソート/パズルで swap/回転列を答える）：action=置換。
- 区間更新ゲーム：同種更新の連続が遅延タグ1個。
- 弱合成のみ：スコアが経路長・累積コストの和（TSP風/スケジューリング）。

## 効かない（はっきり）

- 適応的配置＋連鎖反応（石を置くと連結成分/支配領域が再計算）：閉形式合成なし、
  合成＝手リスト＝歩くのと同じ。
- 連結性・最短路など毎手グローバル再計算が走る状態。
- 鎖が短い／毎手分岐するビーム（位相条件が出ない）。
- **apply がすでに O(1) かつ合成オブジェクトが O(L) の問題**。
  L·a→O(1) が両側から崩れる。

### 15パズル（beam_search/a.cpp）は効かない例

- apply/rollback は memo 済みで O(1)（縮める下が無い）。
- ブランク L 手の合成は support O(L) の盤面置換で、適用も O(L)（畳めない）。
- スコア差分 ±1 は加法的だが既に Action にメモ済み（compose_score 無意味）。
- 生存木は毎手のように分岐（位相条件も弱い）。
- そもそもコスト中心が遷移でなく候補数/選択/メモリ側。

→ apply が O(1) の問題では Euler/LCA 走査が既にほぼ最適。compose 投資は
リターン無し。効くのは「状態評価がインクリメンタル不可で a が重く、解に
長い強制区間がある」逆 regime のとき。

## API 提案 (最小契約)

### Action 側

```cpp
struct Action {
    // ... 既存フィールド ...

    // optional. 実装すると beam_search が forced 鎖を合成して
    // apply 回数を削減できる。
    //
    // 戻り値:
    //   true  : *this を "*this then nxt" の合成に置き換えた。
    //           nxt の中身は consumed (未定義) になる。
    //   false : 合成不可。*this と nxt は両方とも未変更のまま残す。
    //
    // 実装する場合、State::apply_op / rollback / try_op は
    // 合成済み Action を「1 個の Action」として違和感なく扱えること。
    bool compose(Action &nxt);
};
```

### 設計判断とその理由

#### 1. 方向と消費セマンティクスを明文化

- `a.compose(b)` の意味は **`a := a then b`** (前合成)。名前で読めないので
  doc-comment 必須。
- `nxt` は consumed (move されたとみなしてよい)。引数を `Action&&` にすると
  型レベルで伝わるが、tour slot を move したいライブラリ側都合と相性が悪い
  ので `Action&` のまま。
- **失敗時に副作用が無いと約束**しておくと、エンジン側は「失敗したら現在の
  合成を flush して nxt から新しい合成を始める」だけで済む。

#### 2. compose の有無はコンパイル時検出

- SFINAE / `requires` で `Action::compose` の有無を判定。
- 無い場合は現行コード経路 (1 手ずつ apply)。**実行時 branch を増やさない**
  ことが重要 — Action 当たり数 ns のオーバーヘッドでも 15-puzzle 系では
  負ける。
- C++17 縛りなら `void_t` SFINAE で `has_compose<Action>` を生やす。

#### 3. State 側の暗黙契約 (how_to_use に明記すべき)

> `Action::compose` を提供する Action 型に対して、State の
> `apply_op(action)` / `rollback(action)` / `try_op(action, th)` は
> **合成 action を 1 個の action と区別せずに受け付け、合成された手数分の
> 状態変化・スコア差分・hash を 1 回の呼び出しで返す** こと。

これを契約化しておけば、エンジン側は合成済み action と素の action を一様に
扱える。

#### 4. エンジン実装は eager 一択

tour 構造を変えずに走査中に合成する "lazy" 案も考えたが、

- tour エントリは依然として L 個並ぶので iteration コストが減らない
- 毎ターン L 個の compose 蓄積を払い直す
- 救えるのは apply cost だけ (compose ≪ apply のときしか効かない)

と、わざわざ仕組みを足す割に得るものが小さい。素直に **tree 構築時に
forced 鎖を検出して gblock に畳む eager** で入れる。

ただし composed Action が**固定サイズで表現できる前提** (置換・行列・affine
なら OK。可変長な手列を抱えるしかないなら NG)。可変長ケースは gblock の
slot 仕様を弄る必要があり、別途検討。

#### 5. 失敗例 (15-puzzle) は契約で自動除外

15-puzzle のように apply 自体が O(1) かつ Action がメモ済みなら、`compose`
を実装する動機が無い → ユーザは単に compose を生やさない → エンジンは既定
経路 → コストゼロ。

**opt-in が貫けていれば、合わない問題に害は無い。** 現在の素朴な
`struct Action` を維持し続けるユーザにも何も強制しない。

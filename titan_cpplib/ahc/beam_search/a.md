# Action compose（辺縮約）メモ

差分更新ビームサーチの抜本改善案のひとつ。global-LCA より下にできる
**内部一本道**を、1手ずつ apply する代わりに「合成した1手」に畳む。

## 効く条件（2つとも必要）

1. 位相条件：global-LCA より下に*長い内部単一子チェーン*が存在する。
   生存木が「分岐は時々、間は非分岐が長い」形。毎手分岐する問題は対象外。
2. 代数条件：action がモノイドを成し、`compose(a1..aL)` の apply が
   L·a より十分安い。理想 O(1)/O(log)。逆操作も安いこと。

両方そろうと、その鎖の毎世代 2L·a が 2·O(1) に落ちる。

## checkpoint に対する優位

compose は a/s/L の閾値推定が不要。合成オブジェクトは小さく State コピーも
メモリ膨張も無いので「s 払う価値あるか」の判断自体が消える。Action が
compose を出せるなら常に縮約して損しない。閾値問題が無いのが robust な点。

## 段階契約（全部 or 無しにしない）

適用範囲はここで決まる。

- `compose`         : apply 合成可（辺を完全縮約）
- `compose_inverse` : 逆操作も合成可（rollback も得）
- `compose_score`   : スコア差分のみ合成可（移動は1手ずつだが再評価をスキップ）

`compose` が無理でも `compose_score`（スコアが加法的）で拾える問題は多い。

## 抽象例（圧縮できる action）

- 置換：chain = 置換の積 → 1置換。逆も置換。
- アフィン x↦Ax+b：chain = 1アフィン。
- 行列作用 v↦M_a v：chain = 行列積／累乗。
- 区間 add/assign の遅延タグ：タグが合成法則を持つ。
- 加法/カウンタ：chain = 変位の総和（1つの数）。

共通構造：L手の効果を**サイズ一定の1オブジェクト**に畳め、適用も逆も安い。

## 具体例（効く）

- 回廊を動くエージェント：重い状態は位置/占有、分岐は方向のみ。一本道区間は
  単一の平行移動 (dx,dy) に合成。状態移動 O(L)→O(1)。
- オートマトン上の文字列 DP：action=文字=遷移行列。強制文字の連続が行列積1つ。
- 置換列を出力する問題（ソート/パズルで swap/回転列を答える）：action=置換。
- 区間更新ゲーム：同種更新の連続が遅延タグ1個。
- 弱合成のみ：スコアが経路長・累積コストの和（TSP風/スケジューリング）。

## 効かない（はっきり）

- 適応的配置＋連鎖反応（石を置くと連結成分/支配領域が再計算）：閉形式合成なし、
  合成＝手リスト＝歩くのと同じ。
- 連結性・最短路など毎手グローバル再計算が走る状態。
- 鎖が短い／毎手分岐するビーム（位相条件が出ない）。
- **apply がすでに O(1) かつ合成オブジェクトが O(L) の問題**。
  L·a→O(1) が両側から崩れる。

### 15パズル（beam_search/a.cpp）は効かない例

- apply/rollback は memo 済みで O(1)（縮める下が無い）。
- ブランク L 手の合成は support O(L) の盤面置換で、適用も O(L)（畳めない）。
- スコア差分 ±1 は加法的だが既に Action にメモ済み（compose_score 無意味）。
- 生存木は毎手のように分岐（位相条件も弱い）。
- そもそもコスト中心が遷移でなく候補数/選択/メモリ側。

→ apply が O(1) の問題では Euler/LCA 走査が既にほぼ最適。compose 投資は
リターン無し。効くのは「状態評価がインクリメンタル不可で a が重く、解に
長い強制区間がある」逆 regime のとき。

## 実装の差し込み位置（将来）

LCA 走査で一本道を検出し、Action が `compose*` を広告していれば縮約手に
置換。広告が無ければ現行どおり1手ずつ（オプトイン・加算的、既定路線は無傷）。

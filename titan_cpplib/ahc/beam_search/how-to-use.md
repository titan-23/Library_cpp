別途提供するビームサーチのライブラリ実装と対象問題の仕様に基づき、Action構造体とStateクラスを実装してください。

### 1. 実装上の制約と高速化の指針
パフォーマンスの最適化が最重要です。以下の指針に従って実装してください。

- Action構造体の設計とメモリ管理
  - スケルトンコードの`char d`は一例です。対象問題の操作を表現するために必要なメンバ変数（座標やIDなど）に自由に変更・追加してください。
  - Action構造体は大量に生成・コピーされます。ロールバックに必要な情報は「上書きされて失われる前の情報」のみとし、メンバ変数の型は必要十分な最小サイズを選択してサイズを最小化してください。
- スコア計算の打ち切り（最小化問題）
  - 提供されたビームサーチはスコアの最小化を行います。try_op関数に渡されるthresholdは現在のビーム内で最も大きいスコアです。
  - 差分計算の過程でnxt_score >= thresholdになることが確定した時点で、即座に計算を打ち切り、{INF, 0}を返してください。この場合、そのActionがapply_opされることはないため、Actionへのロールバック情報の保存処理もスキップして構いません。
- Zobrist Hashの適切な設計
  - Zobrist Hashによる差分更新を実装してください。ハッシュ化する対象は、盤面の構造など「その後の状態遷移やスコア計算に影響を与える要素」のみに限定し、経過ターン数などは含めないでください。
  - 乱数テーブルを事前に初期化し、XOR演算を用いて高速にハッシュ値を更新してください。
- 状態の差分更新（ディープコピーの禁止）
  - State内の盤面配列などをディープコピーすることは禁止します。必ず変更があった箇所のみを計算・更新してください。
- 不要なActionの枝刈り
  - get_actions関数では、明らかにスコアが悪化する無駄な手や、直前の操作を打ち消すだけの手を生成しないでください。自明な枝刈りを実装してください。

### 2. 各関数の仕様と実装要件
別途提供するスケルトンコード内にある各関数について、以下の要件を満たすように実装してください。関数のシグネチャは絶対に変更しないでください。

#### Action構造体
- 目的: 状態遷移を表す操作パラメータと、ロールバックに必要な情報を保持します。
- 要件: スコアやハッシュの計算結果に加えて、状態を元に戻すための情報をメンバ変数として追加してください。

#### State::init関数
- シグネチャ: void init()
- 目的: 探索開始時の初期状態を構築します。
- 要件: 盤面、スコア、ハッシュ値の初期化を行ってください。Zobrist Hash用の乱数テーブルの初期化もここで行うか、事前に計算済みのものを読み込んでください。

#### State::try_op関数
- シグネチャ: pair<ScoreType, HashType> try_op(Action &action, const ScoreType threshold) const
- 目的: 現在の状態に操作を適用した場合の、次状態のスコアとハッシュ値を計算します。
- 要件: const関数であるためメンバ変数は変更しないでください。スコアとハッシュ値は差分計算で求めてください。計算が最後まで完了した場合のみ、計算結果とロールバックに必要な情報を引数のactionに記録してください。

#### State::apply_op関数
- シグネチャ: void apply_op(const Action &action)
- 目的: 状態を次の状態へ進めます。
- 要件: actionに保存された情報を用いて、Stateのメンバ変数をインプレースで更新してください。※この関数はtry_opを通過した合法なActionに対してのみ呼ばれるため、再度のバリデーションや再計算は不要です。

#### State::rollback関数
- シグネチャ: void rollback(const Action &action)
- 目的: 状態を一つ前の状態に戻します。
- 要件: actionに保存された情報を用いて、apply_op実行前の状態に完全に復元してください。ここでも差分のみを巻き戻してください。

#### State::get_actions関数
- シグネチャ: void get_actions(vector<Action> &actions, const int turn, const Action &last_action) const
- 目的: 現在の状態で実行可能な合法手を列挙します。
- 要件: 可能な操作を生成し、actionsに追加してください。actionsのメモリ再確保を防ぐため、事前に最大生成数がわかる場合はreserveを利用してください。※ターン0の呼び出し時など、last_actionにダミー値が含まれる場合があるため、無条件でlast_actionのメンバを参照して不正アクセスを起こさないよう注意してください。

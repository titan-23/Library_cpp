# AHC典型

## 大方針

- 問題文を言い換える
- 問題を分割する

- 焼きなまし or ビームサーチ
  - 文脈が強いか否か
    - 強くても焼ける場合がある
      - ローリングホライズン
    - 文脈が強そうでも実際はビムサが弱かったりする
      - (AHC046) 基本操作をすると強いので、序盤に操作が固まってしまう
  - 上手く下界を見積もれるとビムサの方が強いことも
  - 制約(盤面)が適度に大きいと、全体を焼く/ビームより、規則的な大域構造を構築で固定する方が強いことがある(探索は規模で薄まるが、構築は規模に対してスケールする)

- 問題を簡単な構造にする
  - グラフ構造への帰着
    - 木構造とみなす
      - 木構造は焼きやすめかも？
      - 最短経路、最小全域木、シュタイナー木
    - DAGとみなす
      - DAGも焼きやすそう
    - 最小費用流、二部マッチング等のグラフアルゴリズム
  - グリッドに限定する(幾何)
  - 初めは粗く、段階的に細かくする

- 典型評価関数
  - 状態を固定したときの最終状態までの最適値
  - 連結成分数、転倒数、LIS、マンハッタン距離、バウンディングボックス、重心、次数、到達可能マス数、分散(密集度)、MST、マッチング、、、、

- 何かが決まれば最適解が求まるか考える
  - その変数を最適化し、その後は厳密(十分良いルールベース)解を適用する

- 最終形の特徴を考える
  - 長時間焼く or レプリカで並列化して良い解を見つける
  - ゴール状態からの逆算
    - 最終的な目標状態から1回の操作、あるいは2回の操作を巻き戻した状態を構築し、その状態が持つ性質を観察する
      - (AHC064) 完成直前の盤面は、複数の正しい連続部分が異なる場所に存在し、それらを連結するだけの状態となるので、連続する部分構造の構築が目標達成の中間目標になることが分かる。
  - 手順を増加させる要因の分析
      - (AHC064) 要素が目標に近い位置にあっても、その前後に不適切な要素が存在すると退避のための余分な操作が発生する。この事実から、要素の絶対的な座標よりも隣接する要素との関係性の方が手数の削減において重要であることが分かる。
  - 手動シミュレーションの言語化
    - 規模の小さい問題を人間が手作業で解き、その際の意思決定の基準を言語化する
      - (AHC064) 人間が操作を行う際、個々の要素を独立して動かすのではなく、要素の塊を構築しそれを単位として動かす戦略を自然に採用します。この無意識に認識している塊という概念を数値化することで、連結成分という特徴を定義できます。
  - 保存すべき構造を考える
      - (AHC064) 探索の過程において、一度構築した後に破壊するべきではない構造を定義します。正しい順序で連結された要素群は、それ以降の操作で分断する論理的な理由がありません。維持するべき構造を定義して評価関数に組み込むことで、構築済みの構造を保護する状態が高い評価を得るようになります。

- 制約緩和
  - **評価関数が難しいのでなるべくやりたくない**
  - 特に、valid でない状態から valid に戻りにくいとき、緩和してペナルティをかけない方が良い気がする

### 一見不向きなアルゴを成立させる工夫

ナイーブな適用では合わないと感じても、解表現や粒度を変えれば成立することが多い。「焼けない / ビムサに乗らない」と切るのは最終手段。

- 焼きなまし
  - 解表現の変換(行動列 → 中継点列 / 順序列 / 決め打ち変数列)
  - 探索空間の制限(明らかに損な構造・対称解の除外)
  - 部分問題への限定(全体は別アルゴ、一部だけ焼く)
  - 評価関数の代替(厳密スコアが困難なら近似 / 下界 / 緩和)
- ビームサーチ
  - 世代の自作(明示的ターンが無くても "1つ決める単位" を自分で定義)
  - 粒度の変換(ミクロ ⇔ マクロ)
  - 評価関数の補強(未来予測 / 下界 / ロールアウト)
  - 部分問題への限定(初期解作りだけビムサ)
- 共通
  - 上記工夫は「最適解の形の予想」と「ある変数を固定すれば残りが決まる分解」から導かれることが多い

### 行き詰まりの兆候(構造へ戻る合図)

- 探索側の改良(評価関数の補強・枝刈り・高速化・パラメタ調整)を2巡しても
  スコアが *構造的に* 頭打ちなら、それは探索の問題ではなく
  **解表現/大域構造の問題のサイン**。改良を続けず 1-D(不変量→レバー)と
  1-C(固定→帰着)へ戻る。
- 評価関数が「効きにくい/何を入れても伸びない」と感じたら、評価関数を直すのでなく
  *評価関数を要らなくする構築* が無いかを 1-C で問い直す。
- 自分が下した診断(例: 「この評価は〇〇を見られていない」)に対し、対策を
  また探索側で打とうとしていないか毎回確認する。診断が構造の欠陥を指すなら
  対策も構造側に打つ。

### ビームサーチ
- 適切な世代を考える
  - (AHC063) 次の一手をとる or 1マス移動
  - 盤面に進行度を保持させる
    - (AHC064) 交差制約のため、一方向の段階的な状態遷移をさせる
  - チューニングすれば基本ミクロな一手の方が強いはず
    - 評価関数が難しいイメージ
- 重複除去
  - 盤面をハッシュ化して探索済み盤面を除去する
  - 厳密には同一盤面でなくても、同一視したいものを除去できれば良い
    - (AHC063) 餌の位置と蛇の頭の位置で十分。蛇の胴体をハッシュ化すると多様性が損なわれる。
- 高速化
  - 自明な下界で閾値との枝刈り(強い)
  - 入力がランダムであることを活かし、一定以下は枝刈ってしまう
- スコアに乱数を入れて多様性確保
- 遅延評価(過去改変)
  - 実際に必要になったときに過去の操作として辻褄を合わせて確定する。

### 焼きなまし
- 解表現を工夫する
  - (AHC046)「1手ごとの行動列」ではなく、「達成すべき中継点と操作のリスト」を状態とする
- 初期解
  - 重要なときとそうでないときがある
- 近傍定義
  - 意味の無い遷移はしない
    - 解が変わる境界のようなものを前計算しておく
  - 具体的な近傍
    - 列
      - 1点挿入, 1点削除, 1点更新, 2点更新(+d,-d)
      - 2点swap, range swap, suffix swap(AHC049)
    - `vector<vector<>>`
      - 1点挿入, 1点削除, 1点更新
      - `vector`内swap, `vector`をまたいだswap
      - 2つ(以上)の`vector`破壊再構築

### 評価関数
- 最終的な目標状態との差分を数値化して定義する
- スコアを悪化させる要因はなにか考える

### 構築

#### ペアで考える(対化)

要素(セル、列、行、ブロック)・エージェント・操作を2つ1組にまとめ、組ごとに役割や基準で分ける。
大域問題を「独立な2要素の小問題の集合」に縮約する構築の常套。

- 狙い / なぜ効くか
  - 結合の局所化: 全体の相互作用を組内に閉じ込め、組同士を疎にする
  - 自由度の確保: 相方を *バッファ/退避/比較器* に使い、大域構造を壊さず操作できる
  - 不変量の利用: 「各要素ちょうど2つに属する(cover2)」「各 X は相異なる Y に隣接」
    等の構造は、そのまま完全マッチング/2彩色として組に落ちる
  - 償却: 安い操作と高い操作を組にしてコストを平均化する

- 効くサイン(認識手がかり)
  - cover ちょうど2 / 次数2 / 2彩色可能 / 二部性 / パリティ制約
  - 「各単位に予備スロットが1つあれば並べ替え・退避が解ける」と感じる
  - バランス制約(和・分散・最大-最小)を満たしたい
  - 操作が隣接2要素の交換(転置)に分解できる

- ペアの作り方(基準は問題依存・複数試す)
  - 大⇔小(両端マッチで和や分散を均す), 多⇔少, 早⇔遅, 重⇔軽
  - 主⇔予備(本体スロット + 隣接の空き = 乗降/退避ランプ)
  - 入⇔出 / 源⇔吸 / 生産⇔消費
  - 近傍/隣接で対にする(幾何), 同種を避けて異種で対にする

- ペア内の役割分担(split)
  - 一方を確定値、他方を調整代(スラック)にする
  - 一方を本流、他方をパーキング(順序を直す退避先)にする
  - 比較器とみなし、組内で大小に整列(ソーティングネットワーク/奇偶転置の単位)

- 代表パターン
  - マッチングで対化 → 残差(奇数個・余り)を別処理。完全マッチの可否を先に確認
  - 主+予備の対化 → 予備を一時退避に使い、本流の順序を探索なしで組み替える
  - 大小ペアリング → 和/分散/最大値を解析的に最適化(両端から貪欲)
  - パリティ/分解 → 奇数次数頂点の対化(オイラー化), ドミノ被覆, 2要素DP

- 組サイズの一般化(2 → 4 → k → ブロック)
  - ペアは k=2 の特殊形。組を 4, 8, …, k と大きくすると
    *組内の自由度(退避枠・並べ替え能力)* が増え、*組あたりコスト/結合* も増える
  - k の選び方: 必要な自由度から逆算。組内を完全に並べ替えたい → k=組丸ごと、
    一部だけ直せれば十分 → 小さい k。両端の中間に最適 k があることが多い(要スイープ)
  - 関連: タイリング/ブロック分割, k進数の桁分解, k-way マージ, k-トーナメント,
    組を1ノードに畳む(状態縮約)
  - まず k=2 で構造と正しさを確認 → k を上げて性能を稼ぐ、の順で進めると安全

- 設計時の問い
  - 対化(組化)の基準は何か(複数候補を列挙し、目的関数に効くものを選ぶ)
  - 完全マッチ/均等分割か。余り(残差)はどこへ吸わせるか
  - 相方(組)は「比較器」「バッファ」「バランサ」のどれとして使うか
  - 組サイズ k はいくつか(必要自由度から逆算し、スイープで詰める)
  - 対化(組化)を固定すると残りは確定するか(→ 1-C の "固定→帰着" と接続)

## プロンプト

各フェーズで前フェーズの発見を引用しながら回答するよう促す。"なんとなく" や "一般論" は禁止し、問題のどの性質から導いたかを必ず明示させる。

### フェーズ1: 問題構造の抽出

#### 1-A: 多重再定式化

```
問題文をストーリーを排し、以下の視点で並行再定義(自然さの評価不要・並列列挙が目的):
1. データ構造(配列/グラフ/グリッド/集合): 許される操作と制約
2. 最適化: 決定変数 / 目的関数 / 制約
3. 既知問題への帰着(TSP/割当/スケジューリング/被覆/区間/フロー等)を最大3つ
4. ゲーム木: 状態 / 行動 / ターン / 終了条件
複数の見え方があれば各々別に提示。
```

#### 1-B: 最適解の形を逆算

```
理論的最適解(または上位想定解)の構造を予想:
- 特徴的な配置 / 順序 / 形状
- 先に決まっているべき変数(決め打てば残りが従属)
- 不変条件 / 対称性
- ランダム解と区別する特徴量
- (必須) 想定上位スコアから「1要素あたり許容操作量」を概算し、必要償却率(1操作が同時に進めるべき要素数)を見積もる。>1 なら要素ごと処理は原理的に不可 → 「1操作で多数要素を同時前進させる構造」を最適解の核心と先に断定し、その構造/操作候補を挙げる(1-D と接続)。

3つ以上、各々 根拠 + 確信度(5段階) + 導出元の問題性質を明示。
```

#### 1-C: 変数の依存分解

```
決定変数を列挙し「ある変数を固定→残りが自明/古典問題に帰着」分解を3つ以上:
例: 順序固定→各手貪欲 / 最終配置固定→最短経路 / 集合固定→DP / 大域構造(レイアウト・資源配置・経路割当)固定→割当・スケジューリング
最低1つは「問題文に無い大域構造そのもの」を固定する分解を含める。
各分解: 固定変数の空間サイズ / 固定後の帰着可否・計算量 / 残りが探索不要で確定するか(閉形式・多項式アルゴ・厳密ルール)。
探索不要で確定する分解は構築解の第一候補として明示。固定すべき変数は後段(焼き解表現/ビムサのノード情報)で再利用。
```

#### 1-D: 操作の代数的性質

```
許される操作集合を分析: 可逆性 / 可換性 / 冪等性 / 生成性(極小生成集合) / 不変量 / 対称性。
各性質を必ず両面で(片面のみは不完全): ①制約=禁止する遷移・到達不能 ②レバー=武器にすると何が「タダ/格安」になる解表現・構造・操作単位か。
不変量・対称性・冪等性は否定形で終えず必ず肯定形(これを保つ構造を設計すれば〜が一括で決まる)に反転。
1操作で多数の決定変数が同時に動くなら、その「1回適用」を探索の1手でなく構築のプリミティブ候補として明示。
出力: 探索空間縮約案 / 枝刈り条件 / 不変量を保つ前提の解表現候補。
```

#### 1-E: 入力構造の確認

```
入力の生成過程・分布から:
- 入力サイズ感(ランダム探索 / 雑な焼き / 貪欲のどれが現実的か)
- 統計的偏り(密度 / クラスタ / 相関)
- ケース難度差を分ける特徴量
- 制約サイズと時間制限から「1状態あたり計算量予算」
→ 全ケース共通戦略で足りるか、ケース分類が要るか判断。
```

#### 1-F: 中間目標

```
初期↔目標の間に置く「中間目標」案を理由付きで。1-B の最適解形と整合させ、各案に:
- 達成後、残りをルールベース/貪欲で詰められるか
- そこから目標へ後戻りが発生しないか
```

#### 1-G: 人間のヒューリスティクスの言語化

```
十分賢い人間が手作業で解くときのヒューリスティクスを、実装可能なルール/評価値に言語化。小例(N=5等)の手動シミュレーションで意思決定基準を抽出するのも可。
```

### フェーズ2: 大方針

#### 2-A: ナイーブ適合度判定

```
フェーズ1を踏まえ、素朴適用の適合度を判定: 1.貪欲 2.ビムサ 3.焼きなまし
判定軸(各軸フェーズ1の記述を引用):
- 途中状態を評価できるか(貪欲/ビムサ前提)
- 解全体を一括保持・意味ある近傍が定義可か(焼き前提)
- 文脈の強さ(序盤が終盤を強く制約=ビムサ向き)
- 後戻り必要性(=焼き向き)
- 下界の見積りやすさ(ビムサ枝刈り強度)
- 基本操作のレバレッジ: 1操作が解/多数変数をどれだけ動かすか。極端に高いと探索は細切れで弱い(1-Dの構築プリミティブと対応)→「その操作を軸に大域構造を構築固定し残りを割当/スケジューリング」を対案として必ず併記。
各案 ○/△/× と判定根拠・最大の壁。
```

#### 2-A': 成立条件の探索

```
2-A で △/× の案に「成立させる工夫」を必ず考える(「無理」は最終手段、安易に切らない)。

- 焼き: 解表現変換(行動列→中継点/順序/集合/決め打ち変数列、1-Cの固定変数列、1-Bの最適形パラメタ化) / 探索空間制限(損・対称解除外) / 部分問題限定 / 評価代替(近似・下界・緩和)
- ビムサ: 世代の自作(1つ決める単位) / 粒度変換(ミクロ⇔マクロ) / 評価補強(未来予測・下界・ロールアウト) / 同一視ハッシュ / 部分問題限定
- 貪欲: 評価階層化(主+タイブレーク+乱択) / 多スタート+上位採用 / マクロ単位貪欲

各工夫: 何を変えると成立するか / 2-A の △× 根拠が解消されるか / 変換コストと情報損失(最適性を失うか) / 再評価 ○/△/×。
混合戦略(ビムサ初期解→焼き等)も具体化。
```

#### 2-B: 最大レバレッジの設計判断

```
2-A' 再評価後の推奨アプローチを1つ選び、最もスコアを左右する設計判断を1つ特定。2〜3案比較しフェーズ1根拠で1案選択。
判断項目例:
- ビムサ: 世代粒度(ミクロ/マクロ、1-B最適形と整合) / ノード情報の同一視粒度(1-C固定変数) / 評価(確定のみ/下界加算/ロールアウト)
- 焼き: 解表現(そのまま/1-C固定変数列/中継点列) / 近傍(地形が階段にならない最小単位) / 評価(厳密/近似/緩和)
- 貪欲: 評価階層化 / 多スタート分散
形式: 判断項目 / 案A・B・C / 長所短所 / 採用案 / 根拠。
```

#### 2-C: 上位想定アプローチ

```
上位一桁は非自明な工夫を1つ以上持つ(特に 2-A' で我々が出せなかった解表現/粒度変換)。候補を3つ:
- 独自解表現(問題文に無い抽象化) / 帰着で計算量削減(1-A) / 操作の代数性で空間縮約(1-D) / 情報量の多い評価(複数ロールアウト平均・二段階) / 入力分布のケース別戦略(1-E) / 最適形からの強制約(1-B) / 一見不向きを成立させる解表現(2-A'延長)
各候補: なぜ効くか(フェーズ1の発見と接続) / 実装難度 / 素朴比のスコア改善オーダー。
最後に最有望を1つ選ぶ。
```

#### 2-D: 最終推奨

```
最終方針を1つ決定:
- 概要(2〜3行) / 状態・解表現(具体) / 行動・近傍 / 評価関数(式or擬似コード) / 期待スコアレンジ / 最大懸念と切替次善策 / 採用しなかった 2-A' 工夫で残す改善余地
方針が外れる最有力シナリオを1つ挙げ、対処も併記。
```

#### 評価関数の特徴量検討

```
この問題の評価関数を考えます。以下の3つの観点から特徴量を提案してください。
- 局所的な盤面の良さ
- 次のターンの操作のしやすさ
- 目的達成の阻害定量化
```

#### 緩和問題による下界設計

```
手数/スコアを最も悪化させる制約・ペナルティを1〜2個特定し、それを完全無視(コスト0・容量∞・ワープ可)した緩和問題を作る。緩和下で現状→目標(最適)への最も単純なロジックと最小手数(最大スコア)の式を出す。
式は「1要素あたり」でなく「1操作が同時に効かせられる最大数(バッチ)あたり」で立てる。レバレッジの高い操作があるのに要素ごとの和で下界を作ると、下界も評価も op 共有を構造的に見落とす。立てた下界と採用予定の評価/ポテンシャルの整合を必ず照合(要素ごとコストなのに真の下界がバッチ単位なら緩和の取り違え)。
この下界を「理想状態までの距離」として評価関数へ組み込む案も提案。
```

# AHC典型

## 大方針

- 問題文を言い換える
- 問題を分割する

- 焼きなまし or ビームサーチ
  - 文脈が強いか否か
    - 強くても焼ける場合がある
      - ローリングホライズン
    - 文脈が強そうでも実際はビムサが弱かったりする
      - (AHC046) 基本操作をすると強いので、序盤に操作が固まってしまう
  - 上手く下界を見積もれるとビムサの方が強いことも

- 問題を簡単な構造にする
  - グラフ構造への帰着
    - 木構造とみなす
      - 木構造は焼きやすめかも？
      - 最短経路、最小全域木、シュタイナー木
    - DAGとみなす
      - DAGも焼きやすそう
    - 最小費用流、二部マッチング等のグラフアルゴリズム
  - グリッドに限定する(幾何)
  - 初めは粗く、段階的に細かくする

- 典型評価関数
  - 状態を固定したときの最終状態までの最適値
  - 連結成分数、転倒数、LIS、マンハッタン距離、バウンディングボックス、重心、次数、到達可能マス数、分散(密集度)、MST、マッチング、、、、

- 何かが決まれば最適解が求まるか考える
  - その変数を最適化し、その後は厳密(十分良いルールベース)解を適用する

- 最終形の特徴を考える
  - 長時間焼く or レプリカで並列化して良い解を見つける
  - ゴール状態からの逆算
    - 最終的な目標状態から1回の操作、あるいは2回の操作を巻き戻した状態を構築し、その状態が持つ性質を観察する
      - (AHC064) 完成直前の盤面は、複数の正しい連続部分が異なる場所に存在し、それらを連結するだけの状態となるので、連続する部分構造の構築が目標達成の中間目標になることが分かる。
  - 手順を増加させる要因の分析
      - (AHC064) 要素が目標に近い位置にあっても、その前後に不適切な要素が存在すると退避のための余分な操作が発生する。この事実から、要素の絶対的な座標よりも隣接する要素との関係性の方が手数の削減において重要であることが分かる。
  - 手動シミュレーションの言語化
    - 規模の小さい問題を人間が手作業で解き、その際の意思決定の基準を言語化する
      - (AHC064) 人間が操作を行う際、個々の要素を独立して動かすのではなく、要素の塊を構築しそれを単位として動かす戦略を自然に採用します。この無意識に認識している塊という概念を数値化することで、連結成分という特徴を定義できます。
  - 保存すべき構造を考える
      - (AHC064) 探索の過程において、一度構築した後に破壊するべきではない構造を定義します。正しい順序で連結された要素群は、それ以降の操作で分断する論理的な理由がありません。維持するべき構造を定義して評価関数に組み込むことで、構築済みの構造を保護する状態が高い評価を得るようになります。

- 制約緩和
  - **評価関数が難しいのでなるべくやりたくない**
  - 特に、valid でない状態から valid に戻りにくいとき、緩和してペナルティをかけない方が良い気がする

### 一見不向きなアルゴを成立させる工夫

ナイーブな適用では合わないと感じても、解表現や粒度を変えれば成立することが多い。「焼けない / ビムサに乗らない」と切るのは最終手段。

- 焼きなまし
  - 解表現の変換(行動列 → 中継点列 / 順序列 / 決め打ち変数列)
  - 探索空間の制限(明らかに損な構造・対称解の除外)
  - 部分問題への限定(全体は別アルゴ、一部だけ焼く)
  - 評価関数の代替(厳密スコアが困難なら近似 / 下界 / 緩和)
- ビームサーチ
  - 世代の自作(明示的ターンが無くても "1つ決める単位" を自分で定義)
  - 粒度の変換(ミクロ ⇔ マクロ)
  - 評価関数の補強(未来予測 / 下界 / ロールアウト)
  - 部分問題への限定(初期解作りだけビムサ)
- 共通
  - 上記工夫は「最適解の形の予想」と「ある変数を固定すれば残りが決まる分解」から導かれることが多い

### ビームサーチ
- 適切な世代を考える
  - (AHC063) 次の一手をとる or 1マス移動
  - 盤面に進行度を保持させる
    - (AHC064) 交差制約のため、一方向の段階的な状態遷移をさせる
  - チューニングすれば基本ミクロな一手の方が強いはず
    - 評価関数が難しいイメージ
- 重複除去
  - 盤面をハッシュ化して探索済み盤面を除去する
  - 厳密には同一盤面でなくても、同一視したいものを除去できれば良い
    - (AHC063) 餌の位置と蛇の頭の位置で十分。蛇の胴体をハッシュ化すると多様性が損なわれる。
- 高速化
  - 自明な下界で閾値との枝刈り(強い)
  - 入力がランダムであることを活かし、一定以下は枝刈ってしまう
- スコアに乱数を入れて多様性確保
- 遅延評価(過去改変)
  - 実際に必要になったときに過去の操作として辻褄を合わせて確定する。

### 焼きなまし
- 解表現を工夫する
  - (AHC046)「1手ごとの行動列」ではなく、「達成すべき中継点と操作のリスト」を状態とする
- 初期解
  - 重要なときとそうでないときがある
- 近傍定義
  - 意味の無い遷移はしない
    - 解が変わる境界のようなものを前計算しておく
  - 具体的な近傍
    - 列
      - 1点挿入, 1点削除, 1点更新, 2点更新(+d,-d)
      - 2点swap, range swap, suffix swap(AHC049)
    - `vector<vector<>>`
      - 1点挿入, 1点削除, 1点更新
      - `vector`内swap, `vector`をまたいだswap
      - 2つ(以上)の`vector`破壊再構築

### 評価関数
- 最終的な目標状態との差分を数値化して定義する
- スコアを悪化させる要因はなにか考える

## プロンプト

各フェーズで前フェーズの発見を引用しながら回答するよう促す。"なんとなく" や "一般論" は禁止し、問題のどの性質から導いたかを必ず明示させる。

### フェーズ1: 問題構造の抽出

#### 1-A: 多重再定式化

```
問題文を、ストーリー要素を排して以下の "複数の視点" から並行して再定義してください。
1. データ構造視点(配列 / グラフ / グリッド / 集合)で、許される操作と満たすべき制約は何か
2. 最適化視点で、決定変数 / 目的関数 / 制約は何か
3. 既知問題への帰着視点(TSP / 割当 / スケジューリング / 被覆 / 区間スケジューリング / フロー 等)で、近いものを最大3つ
4. ゲーム木視点で、状態 / 行動 / ターン / 終了条件は何か

同じ問題が複数の見え方を持つ場合、それぞれを別々に提示してください。
"どの見え方が一番自然か" の評価は不要です。並列に並べることが目的です。
```

#### 1-B: 最適解の形を逆算

```
仮にこの問題の理論的最適解(または想定上位陣の解)が手元にあるとして、
それはどのような構造を持っていると予想されますか?

- 特徴的な配置 / 順序 / 形状はあるか
- 最適解で "先に決まっている" べき変数は何か(これを決め打てば残りが従属する)
- 最適解が満たす不変条件 / 対称性は何か
- ランダムな解と最適解を区別する特徴量は何か

3つ以上の予想を、それぞれ根拠とともに挙げてください。
問題のどの性質から導いたかを明示し、各予想に確信度を 5段階 で併記してください。
```

#### 1-C: 変数の依存分解

```
この問題の決定変数を列挙し、
"ある変数を固定すると、残りの変数が自明 / 古典問題に帰着する"
という分解パターンを3つ以上探してください。

例:
- 順序を固定 → 各ステップの選択が貪欲で決まる
- 最終配置を固定 → そこへの遷移が最短経路
- 集合を固定 → 内部の最適化が DP

各分解について、
- 固定する変数の探索空間サイズ
- 固定後の問題の解きやすさ(古典問題への帰着可否、計算量)

を評価してください。
ここで見つけた "固定すべき変数" は、後段で焼きなましの解表現候補・ビムサのノード情報候補として再利用します。
```

#### 1-D: 操作の代数的性質

```
許される操作集合について以下を分析してください。
- 可逆性: 逆操作の存在と手数
- 可換性: 操作の順序を入れ替えても結果が同じか
- 冪等性: 同じ操作の繰り返しの挙動
- 生成性: 基本操作の組み合わせで全状態空間を覆えるか、極小生成集合は何か
- 不変量: 操作で変わらない値・パリティ・関係
- 対称性: 状態間で同一視できる対称変換

これらから導かれる "探索空間の縮約" 案と "枝刈りの条件" を提案してください。
```

#### 1-E: 入力構造の確認

```
入力データ(N、配置、グラフ等)の生成過程・分布から以下を読み取ってください。
- 統計的偏り(密度 / クラスタ / 相関)
- ケースごとの難度差(易しいケースと難しいケースを区別する特徴量)
- 制約サイズと実行時間制限から逆算した "1状態あたりに使える計算量予算"

これらから "全ケース共通の戦略で十分か、ケース分類が必要か" を判断してください。
```

#### 1-F: 中間目標

```
初期状態と目標状態の間に置くべき "中間目標" のアイデアを、理由とともに挙げてください。
1-B の "最適解の形" との整合性を意識し、
- それが達成された後、残りはルールベース / 貪欲で詰められるか
- その中間状態から目標へ後戻りが発生しないか

を併記してください。
```

#### 1-G: 人間のヒューリスティクスの言語化

```
十分賢い人間がこの問題を手作業で解くとしたら、どのようなヒューリスティクスで最適解に近づけると思いますか?
人間が直感的に行っているその判断基準を、アルゴリズムとして実装できるルールや評価値に言語化してください。
小さい例(N=5 等)で手動シミュレーションを試み、その意思決定基準を抽出するのも有効です。
```

### フェーズ2: 大方針

#### 2-A: ナイーブ適合度判定

```
フェーズ1の発見を踏まえ、以下3アプローチを "問題文の解をそのまま持つ素朴な適用" で判定してください。

1. 貪欲法(評価関数を1回適用、後戻りなし)
2. ビームサーチ(複数状態を保持、世代ごとに上位を残す)
3. 焼きなまし(完成解を近傍遷移で改善)

判定軸(各軸についてフェーズ1の具体的記述を引用すること):
- 途中状態の良さを評価できるか(ビムサ / 貪欲の前提)
- 解全体を一括で持てるか、意味ある近傍が定義できるか(焼きなましの前提)
- 文脈の強さ: 序盤の選択が終盤を強く制約するか(ビムサ向きかの判断)
- 局所最適の罠: 後戻りの必要性(焼きなまし向きかの判断)
- 良い下界の見積もりやすさ(ビムサの枝刈り強度)

各アプローチを ○ / △ / × で判定し、判定根拠と最大の壁を明記してください。
```

#### 2-A': 成立条件の探索

```
2-A で △ または × と判定したアプローチについて、"成立させるための工夫" を必ず考えてください。
"工夫しても無理" は最終手段であり、安易に切らないでください。

[焼きなましを成立させる工夫]
- 解表現の変換: 問題文の解そのままではなく別の表現に変える
  - 行動列 → 中継点列 / 順序列 / 集合 / 決め打ち変数列
  - フェーズ1-C で見つけた "固定すべき変数" の列を解にする
  - フェーズ1-B の "最適解の形" を満たすパラメタ化を解にする
- 探索空間の制限: 明らかに損な構造・対称な解を初めから除外
- 部分問題への限定: 全体は別アルゴ、一部だけ焼く
- 評価関数の代替: 厳密スコアが計算困難なら近似 / 下界 / 緩和

[ビームサーチを成立させる工夫]
- 世代の自作: 問題文に明示的ターンが無くても "何かを1つ決める単位" を定義
- 粒度の変換: ミクロ(1手) ⇔ マクロ(意味ある操作列)
- 評価関数の補強: 途中評価が困難なら 未来予測 / 下界 / ロールアウト で補う
- 同一視ハッシュ: 状態空間が大きいなら本質情報のみ持つ
- 部分問題への限定: 初期解作りだけビムサ、改善は別アルゴ

[貪欲を成立させる工夫]
- 評価関数の階層化(主評価 + タイブレーカ + 乱択)
- 多スタート + 上位採用
- マクロ操作単位での貪欲

各工夫について:
- 何を変えると成立するか(具体的に)
- その変換でフェーズ1の壁(2-A で指摘した △ / × の根拠)が解消されるか
- 変換コスト(実装難度) と "変換による情報損失"(最適性を失わないか)
- 変換後の適合度を ○ / △ / × で再評価

混合戦略(ビムサで初期解 → 焼きなまし、等)もこの段階で具体化してください。
```

#### 2-B: 最大レバレッジの設計判断

```
2-A' で再評価後の適合度を踏まえ、推奨アプローチを1つ選び、
そのアプローチで "最もスコアを左右する設計判断" を1つ特定してください。
判断項目には複数の選択肢があるはずなので、2〜3案を比較し、
フェーズ1の発見を根拠に1案を選んでください。

判断項目の例:
[ビムサの場合]
- 世代の粒度: ミクロ / マクロ → フェーズ1-B の "最適解の形" と整合する粒度はどちらか
- ノードに持つ情報(同一視の粒度): フェーズ1-C の "固定すべき変数" を持つか
- 評価関数: 確定スコアのみ / 残り手数の下界を加える / ロールアウト

[焼きなましの場合]
- 解表現: 問題文の形そのまま / フェーズ1-C で見つけた "固定変数" 列 / 中継点列
- 近傍: スコア地形が階段にならない最小単位は何か
- 評価関数: 厳密スコア / 近似 / 緩和ペナルティ

[貪欲の場合]
- 評価関数の階層化(主評価 + タイブレーカ)
- 多スタートの分散

回答は "選んだ判断項目 / 案A・B・C / それぞれの長所短所 / 採用案 / 採用根拠" の形式で。
```

#### 2-C: 上位想定アプローチ

```
このコンテストで上位一桁に入る人は、おそらく我々がまだ思いついていない非自明な工夫を1つ以上持っています。
特に "2-A' で挙げた成立条件のうち、我々が思いつかなかった解表現や粒度変換" が
上位陣の差別化要因である可能性が高いです。

候補を3つ挙げてください。
- 問題構造を利用した独自の解表現(問題文の言葉では出てこない抽象化)
- 別問題への帰着で計算量を一気に削る(フェーズ1-A の帰着候補から)
- 操作の代数的性質を利用した探索空間縮約(フェーズ1-D から)
- 情報量の多い評価関数(複数ロールアウト平均、二段階評価)
- 入力分布を利用したケース別戦略(フェーズ1-E から)
- 最適解の形からの逆算で得られる強い制約(フェーズ1-B から)
- 一見不向きなアルゴを成立させるための非自明な解表現(2-A' の延長)

各候補について:
- なぜ効きそうか(フェーズ1のどの発見と結びつくか)
- 実装難度
- 素朴な実装に対するスコア改善幅の期待値(オーダー感)

最後に "どれを試す価値が最も高いか" を選んでください。
```

#### 2-D: 最終推奨

```
ここまでの分析を踏まえ、最終的に取る大方針を1つに決めてください。
- 概要(2〜3行)
- 状態 / 解表現の定義(具体的に)
- 行動 / 近傍の定義
- 評価関数(具体式または擬似コード)
- 期待スコアレンジの粗い見積もり
- 最大の懸念点と "懸念が顕在化したときに切り替える次善策"
- 採用しなかった 2-A' の工夫のうち、改善余地として残しておくもの

この方針が外れる最も可能性の高いシナリオを1つ自分で挙げ、その場合の対処も併記してください。
```

### 補助プロンプト(大方針確定後、必要に応じて)

#### 評価関数の特徴量検討

```
この問題の評価関数を考えます。以下の3つの観点から特徴量を提案してください。
- 局所的な盤面の良さ
- 次のターンの操作のしやすさ
- 目的達成の阻害定量化
```

#### 緩和問題による下界設計

```
この問題において、手数を増加させている(またはスコアを悪化させている)最も厄介な制約やペナルティを1〜2個特定し、
それらを "完全に無視してよい(コスト0、容量無限大、ワープ可能など)" とした緩和問題を考えてください。
その緩和問題において、現在の状態から目標状態(または最適解)へ到達するための最も単純なロジックと、
その際にかかる最小手数(または最大スコア)の計算式を答えてください。
さらに、その計算結果を元の問題における "理想状態までの距離(下界)" として、
評価関数に組み込むアイデアを提案してください。
```

# フレームワークの全体仕様と目的
このテンプレートは外部の実行側コード（sa.cpp）から読み込まれ、焼きなまし法で動作します。

【厳守事項】
- 外部フレームワークとの連携のため、テンプレートにある `#include`、`namespace sa` の枠組みは絶対に改変しないでください。
- クラス名（`State`, `Changed`）、グローバル変数名（`changed`, `sarnd`）、各メソッドのシグネチャは変更しないでください。
- `State` および `Changed` のデフォルトコンストラクタ（`State() {}`, `Changed() {}`）は配列の初期化に必須なため、絶対に削除しないでください。
- スレッドセーフを破壊するため、グローバル変数や関数内の `static` 変数を新たに定義することは厳禁です。キャッシュや作業配列が必要な場合は、`State` のメンバ変数か `Changed` 構造体の中に定義してください。

## クラス・変数の目的
- `State` クラス: 1つの状態（盤面・解）の「現在の状態」と「現在のスコア」を保持します。
- `Changed` 構造体 (`thread_local`): 1手分の「遷移の差分情報（どこを、何から、何へ変更したか）」や「変更前のスコア」を記録する作業領域です。
  - 【必須要件】メンバ変数 `TYPE_CNT` には、今回実装する近傍操作の種類の総数を必ず設定・上書きしてください
- `sarnd` (`thread_local titan23::Random`): 固有の高速なXorshift乱数生成器です。以下の仕様に従って使用してください。
  - `sarnd.random()`: [0.0, 1.0] の実数を返す。
  - `sarnd.randint(end)`: [0, end] の整数を返す（両端含む）。
  - `sarnd.randint(begin, end)`: [begin, end] の整数を返す（両端含む）。
  - `sarnd.randrange(end)`: [0, end) の整数を返す（endを含まない）。
  - `sarnd.randrange(begin, end)`: [begin, end) の整数を返す（endを含まない）。
  - `sarnd.shuffle(vec)`: vectorをインプレースにシャッフルする。

## 処理のライフサイクル（実行フロー）
1. 初期化: 各スレッド・各レプリカで1回だけ `State::init()` が呼ばれる。
2. 近傍遷移ループ: 制限時間まで以下を繰り返す。
   1. `state.reset_is_valid()` : 有効フラグをリセット。
   2. `state.modify(threshold, progress)` : 近傍操作を選択し、新スコアを計算。変更履歴を `changed` に記録。
   3. 実行側でスコアを評価し、採否を決定。
      1. 採択 (Accept) の場合: `state.advance()` が呼ばれる。
      2. 棄却 (Reject) の場合: `state.rollback()` が呼ばれ、`changed` の情報をもとに状態を `modify` を実行する前の状態へ巻き戻す。


# 各メソッドの正確な仕様と実装ルール

## 1. 事前準備
- `sa_init()`:
  プログラム全体で最初に1回だけ呼ばれます。乱数以外の事前計算があればここに実装してください。

## 2. 初期解の生成
- `State::init()`:
  状態・配列の初期化を行い、初期解のスコアを計算してメンバ変数 `score` に代入してください。

## 3. 状態の遷移
- `State::modify(const ScoreType threshold, const double progress)`:
  - 引数 `progress` は 0.0（開始時）から 1.0（終了時）へ推移する時間の進行度です。必要に応じて近傍の選択確率や変化量の調整に使用してください。
  - 乱数 `sarnd` を用いて近傍操作を1つ選び、新しい `score` を計算してください。盤面を直接更新するか、スコア差分だけ計算する（遅延評価）かは自由です。
  - 実行した操作の種類を `changed.type` に 0-indexed の整数（0 から TYPE_CNT-1 の範囲） で必ず代入してください。範囲外の値を代入すると実行側で配列外参照エラーになります。
  - 後で巻き戻せるよう、変更対象のインデックス、元の値、元のスコアなどを `changed` に保存してください。`changed` は `thread_local` であり前回のループのゴミデータが残っているため、使用するメンバ変数は今回必ず上書き更新してください。
  - 操作がルール違反となる場合は `is_valid = false;` とし、即座に `return` してください。

## 4. 遷移の巻き戻しと確定
- `State::rollback()`:
  - 遷移が棄却された場合に呼ばれます。`modify` で盤面の状態を直接更新した場合は、`changed` に保存した情報を用いて状態変数と `score` を完全に元の状態に戻してください。
- `State::advance()`:
  - 遷移が採択された場合に呼ばれます。`modify` でスコア計算だけを行う「遅延評価」を採用した場合は、ここで状態変数の書き換えを確定させてください。`modify` で既に状態を更新済みの場合は空のままで構いません。

## 5. スコアの扱い
- テンプレート内の `using ScoreType = double;` の部分は、問題のスコア計算に最適な型（`long long` や `double` など）に適切に変更してください。
- 最小化問題として実装: 実行側はスコアが小さいほど良いとして動作します。問題が最大化の場合は、計算したスコアにマイナスをつけて `score` 変数に代入してください。
- 二重反転の禁止: 最大化問題で `score` 変数にマイナスをつけて保持する場合、`get_score()` の中身は単に `return score;` のままにしてください（ここでさらにマイナスをつけると最小化フレームワークが崩壊します）。
- `get_true_score()`: 問題文で定義されている本来の生のスコアを返すように実装してください。


# 効率よく高速なコードを書くためのヒント
- 差分更新: `modify` 内で盤面全体を再計算したり、配列全体をコピーしたりしないでください。変更が影響する局所的な範囲のみを再計算し、スコアの差分を加減算してください。
- Changed構造体の活用: `changed` は都度インスタンス化されないため、変更記録や一時計算用の「大きな配列・バッファ」を持たせてもパフォーマンス上の問題はありません。ループ内での動的メモリ確保を避けるための作業領域として積極的に活用してください。
- 不要な計算の打ち切り: `modify` の中で差分計算を行っている途中に、スコアの悪化量が `threshold` を明らかに超えることが判明した場合は、その時点で計算を打ち切り `is_valid = false` にして `return` してください。ただし、この際に盤面を中途半端に書き換えていると `rollback()` 時に不整合が起きるため、打ち切りを行う場合は「盤面を書き換える前に遅延評価で判定する」実装を強く推奨します。

以上の仕様、ルール、およびヒントに厳格に従い、出力してください。
なお、現状replica_runは使用しないため、スレッドセーフに関してのみ考慮しなくて良いとします。
